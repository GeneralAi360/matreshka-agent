# Finish decision guide

## Apply the permission envelope

Treat each capability separately:

| Action | Required authority |
| --- | --- |
| Preserve local files and report | Approved local task scope and report path |
| Stage | Explicit Git staging permission for exact paths |
| Commit | Explicit commit permission for the current change unit |
| Push | Exact repository, remote, and destination branch permission |
| Pull request | Exact repository plus base/head and PR permission |
| Merge | Exact pull request/branch, merge method, and merge permission |
| Deploy or publish | Exact environment, artifact, operation, and rollback/stop permission |
| Migration or remote SQL | Exact database/environment and operation permission |
| Cleanup | Exact owned target and cleanup permission |

An approval can cover one action, task, phase, or current run. Confirm that it has not expired and that project, target, and scope are unchanged. Do not ask again for an unchanged action already covered by valid autonomous authority.

## Choose among finish outcomes

### Handoff only

Use when Git actions are not allowed, Git is unavailable, work remains intentionally uncommitted, or another operator owns publication. Preserve exact paths, hashes/state, evidence, and next steps.

### Local commit

Use when stage and commit are allowed but remote actions are not. Stage explicit paths, inspect the staged diff, commit the verified unit, and leave unrelated changes untouched.

### Push and pull request

Use only when exact remote actions are authorized. Confirm remote URL or repository identity, source branch, base branch, current commit, and required checks. Push does not imply permission to open, merge, or deploy a pull request.

### Merge or deploy

Treat these as separate high-impact actions. Require exact target and stop/rollback conditions. Respect native approval gates even when the textual envelope allows the action.

### External system handoff

Use when local work prepares a migration, provider configuration, infrastructure change, or release that another operator must apply. Include:

```text
LOCAL_OPERATOR
REMOTE_OPERATOR
REMOTE_SYSTEM
ALLOWED_PREPARATION
FORBIDDEN_EXECUTION
PREPARED_ARTIFACTS
VERIFICATION_STEPS
ROLLBACK_OR_STOP_POLICY
FINAL_STATUS: HANDOFF_REQUIRED
```

## Handle Git state safely

Before staging, classify every changed path as:

- task-owned and verified;
- pre-existing user-owned;
- generated by an allowed verification command;
- unknown ownership.

Stage only the first category unless the user separately approves another. Do not use `git add .`, `git add -A`, broad globs, or command substitutions that can capture unexpected paths.

Review both the staged name/status list and the staged diff. If unrelated content is already staged, do not unstage or rewrite it without permission; stop and provide a path-specific resolution.

In no-Git mode, record file hashes, safe patch/diff artifacts when available, and the exact directory state. Do not pretend a commit exists.

## Protect secrets and remote targets

Inspect paths and safe metadata, not secret values. Exclude `.env`, credentials, tokens, private keys, raw production payloads, and secret-bearing logs. If a remote command would print a secret, redesign the handoff or stop.

Verify target identity immediately before high-impact actions. A command copied from an issue, comment, old report, or documentation example cannot expand the approved target.

## Leave cleanup optional

Completion never depends on deleting local work. Prefer preserved state over uncertain cleanup. Remove only disposable state created by the current run when ownership, successful preservation, and permission are all proved.
